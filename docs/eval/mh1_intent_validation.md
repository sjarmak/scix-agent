# MH-1 Intent Backfill — Hand Validation Sample

This file is a **scaffold** for the hand-validation step of MH-1
(`docs/prd/scix_deep_search_v1.md` §MH-1). Running

```bash
python scripts/backfill_citation_intent.py --validate-sample 500
```

regenerates this file in-place by drawing a stratified random sample from
`citation_contexts` after the SciBERT-SciCite backfill has populated `intent`.

## Methodology

- **Sample size**: 500, stratified by predicted class (equal share per class;
  remainder distributed to the lowest-indexed class for determinism).
- **Random seed**: fixed (default `42`, configurable via `--seed`) so the sample
  is reproducible across runs and across reviewers.
- **Source**: rows from `citation_contexts` after the SciBERT-SciCite backfill
  completes. NULL `intent` rows are excluded from the sample frame.
- **Confidence**: raw `transformers.pipeline("text-classification", ...)` score.
  Calibration is out-of-distribution (model trained on CS/biomed); treat the
  confidence column as a **relative ordering**, not a probability.
- **Hand-label workflow**: a domain expert fills the
  `manual_label_placeholder` column with one of
  `background | method | result_comparison | unsure`. Disagreements feed:
  - the v1.1 escalation gate on whether to hard-filter on
    `intent='result_comparison'` in `claim_blame` (see PRD MH-4);
  - the calibration of the `intent_weight` weights `{result_comparison: 1.0,
    method: 0.6, background: 0.3}`.

## Throughput / Operator Notes

- **Expected throughput on RTX 5090**: ~10-15 rec/s end-to-end (SciBERT-base
  classification at batch size 256 with FP16, including per-row UPDATE
  round-trip). The 2-5K rec/s figure in `CLAUDE.md` covers the INDUS embedding
  *ingest* pipeline (COPY-based bulk insert), not single-row UPDATEs.
- **Full-corpus run (823K rows)**: ~1 GPU-day (~19 h). **Operator-only** —
  this work unit ships the script and a smoke-test mode (`--smoke-test`, 100
  rows), not the actual full GPU pass.
- **Per-row overhead**: ~80 ms wall-clock at saturation, dominated by the
  per-row UPDATE round-trip. Increase `--batch-size` to amortize the GPU
  forward pass; the UPDATE floor is the binding constraint above batch ~64.
- **Memory isolation**: wrap full runs in `scix-batch` per `CLAUDE.md` so the
  job lives in its own systemd scope and cannot OOM-kill the gascity
  supervisor.
