## Context — what M4 tested and what it found

The parent PRD's M4 requirement was a three-way retrieval eval on the populated 32.4M-paper corpus:

- **Lane A — hybrid baseline.** Existing hybrid search (INDUS + text-3-large + BM25 via RRF).
- **Lane B — hybrid + static-core filter (M6 default path).** Candidates restricted to bibcodes present in `document_entities_canonical` (the M6 curated-core materialized view, ~38% corpus coverage).
- **Lane C — hybrid + JIT re-rank (M12 JIT path).** Candidates re-ranked by `fused_confidence × bibcode_entity_overlap`.

Ground truth was citation-based, drawn from 50 seed queries + 20 agent graph-walk tasks against the real 32M/16.3M-entity-link state. Results (from bead `scix_experiments-xz4.1.10` Decision note; full numbers in `build-artifacts/m4_inhouse_eval.md`):

| Lane | nDCG@10 | Recall@20 | MRR    | Δ nDCG vs baseline |
| ---- | ------- | --------- | ------ | ------------------ |
| Hybrid baseline        | 0.1845 | 0.0862 | 0.4141 | —       |
| Hybrid + static-core   | 0.1582 | 0.0680 | 0.3436 | **−14%** |
| Hybrid + JIT re-rank   | 0.1641 | 0.0752 | 0.4079 | **−11%** |

The static-core lane regressed on every metric (nDCG −14%, Recall −21%, MRR −17%). Restricting candidates to the entity-linked subset drops relevant papers that live outside the covered 38%. The JIT re-rank lane likewise underperformed baseline (−11% nDCG / −13% Recall / −1.5% MRR); on the full 70-item set combined with 20 graph-walk tasks, JIT converges within noise (−0.003 nDCG), so its best claim is parity with baseline — no measurable improvement.

The M4.5 lane-consistency gate also failed (p90 adjusted Jaccard = 0.6667 vs 0.05 threshold), but that failure reflects semantic divergence between lanes (citation-chain neighbors carry topic-adjacent entities, not seed-identical), not an implementation bug. Gate semantics are being re-worked separately in bead `scix_experiments-xz4.1.20` and do not factor into this amendment.

## Decision — M6 and M12 removed from the default retrieval rollout

Per the parent PRD's preserved-dissent clause ("if the M4 three-way eval shows the static lane adds no benefit beyond `paper_uat_mappings` + JIT, B's position wins ex post and M6/M12 collapse to JIT-only"), Advocate B's dissent wins.

**Struck from the rollout plan:**

1. **M6 as a default retrieval pre-filter.** Restricting hybrid-search candidates to bibcodes present in the curated-core MV is no longer part of the default retrieval path.
2. **M12 static-core default path** — the dual-lane routing's default-on promotion of static-core entity-overlap rerank into retrieval tools. The dual-lane **architecture** (M12) remains; its **default-on retrieval behavior** is removed.

**Retained:**

- M1, M2, M3, M3.5, M4, M5, M8, M9, M10, M11a/b/c, M13 — unchanged.
- The curated-core materialized view (`document_entities_canonical`) itself continues to exist and is populated. It remains valuable for graph-analytics tooling (`citation_chain`, `co_citation_analysis`, `bibliographic_coupling`, community detection, MCP-side diagnostics) where extracting on-demand against 299M citation edges is infeasible.
- M13 (`resolve_entities()` service) and its lane-invariance property test.
- All S-tier and N-tier items, including S4 ontology version pinning, S5 feedback loop, and N2 Wikidata backfill.

## Scope of this decision — what it does NOT claim

The M4 eval tested two specific, narrow implementations of entity enrichment:

- **hard-filter to tagged papers** (static-core pre-filter), and
- **naive entity-overlap rerank** (JIT static-core path).

Both are strawman lanes relative to the mechanisms by which entity enrichment is expected to help retrieval. The following value props were **never evaluated** by M4 and are **not touched** by this amendment:

1. **Alias-driven query expansion** (HST → Hubble Space Telescope) — expected recall lift.
2. **Ontology-metadata expansion** via entity properties JSONB (M-type asteroids → Psyche) — neither dense nor BM25 can resolve this.
3. **Entity disambiguation** at query time (Hubble the person vs Hubble the mission; Psyche the asteroid vs the spacecraft) — precision lift.
4. **Entity-type filters** (agent-driven "only instruments" toggle) — interactive control.
5. **Mention-specific filters** (papers that discuss Psyche-the-asteroid vs papers containing the token "psyche") — precision lift.
6. **Community/cluster expansion** via Leiden communities — recall across vocabulary gaps.

**Correct framing:** *default-on static-core promotion hurts retrieval; entity-graph value remains to be evaluated in its actual use modes.*

**Incorrect framing to avoid:** "entity enrichment doesn't help retrieval." M4 did not support that claim.

## Paper-writeup framing guidance

Ready-to-paste paragraph for Section 5 or 6 of the paper:

> A three-way retrieval eval on 32.4M papers and 16.3M entity links (50 citation-seeded queries plus 20 agent graph-walk tasks) found that promoting the static entity-linked subset as a default retrieval pre-filter regressed nDCG@10 by 14%, Recall@20 by 21%, and MRR by 17% relative to hybrid RRF (INDUS + text-embedding-3-large + BM25). A naive entity-overlap JIT re-rank was likewise indistinguishable from baseline. These lanes are strawman uses of the entity graph: they test whether coverage alone is enough to move retrieval, and it is not, because restricting candidates to tagged papers drops relevant documents outside the ~38% entity-linked subset. The richer value props — query-time alias expansion, ontology-metadata expansion via structural properties, sense-disambiguation for homographs, entity-type filters, and community-based expansion — were not exercised by this eval and are reported separately. The architectural conclusion is narrow: default-on static-core promotion hurts retrieval on a large heterogeneous corpus; the graph's value for retrieval lives in query-shaped mechanisms that use structure, not in gating the candidate set on membership.
